# 설명 편집기·리치 에디터 로드맵

현재 설명 필드는 **마크다운 텍스트 영역 + GFM 미리보기**로 동작한다. 아래는 향후 **WYSIWYG/리치 텍스트**를 도입할 때 참고할 전제와 선택지 정리다.

## 유지할 것

- **저장 형식**: 마크다운(또는 마크다운과 양방향 변환 가능한 AST)을 단일 소스로 두는 것이 DB·동기화·diff·이전 버전과 맞다.
- **GFM**: 표, 체크리스트, 취소선 등은 `remark-gfm` / `mdast` 생태계와 맞춘다.
- **작업/북마크 링크**: `todox-task:` 등 커스텀 스킴은 파서/직렬화 시 보존 규칙을 문서화한다.

## 라이브러리 후보(비교용)

| 방향 | 예시 | 장단 |
|------|------|------|
| ProseMirror / Tiptap | Tiptap, ProseMirror schema | 스키마로 제한·확장이 강함, 마크다운 import/export는 별도 파이프라인 |
| Lexical | Meta Lexical | React 친화, 마크다운은 공식이 아닌 커스텀 직렬화 |
| BlockNote | blocknote | 노션 유사 블록, 마크다운 완전 호환은 추가 작업 |
| 유지: textarea + 미리보기 | 현재 | 가볍고 예측 가능, 리치 편집은 미리보기·툴바로 보완 |

## 마크다운 호환 이슈

- **빈 줄·들여쓰기**: 리치 편집기에서 붙여넣기 시 코드 블록/인용이 깨지지 않게 정규화 규칙이 필요하다.
- **체크리스트**: 소스 줄 인덱스와 AST 순서가 중첩 목록에서 어긋날 수 있어, 동기화가 필요하면 **mdast 기준 인덱스**를 쓰는 편이 안전하다.
- **실행 취소**: 에디터 전환 시 히스토리 스택을 하나로 유지할지, 편집 모드별로 둘지 결정한다.

## 제안되는 다음 단계

1. 사용자 요구(모바일 키보드, 협업, 주석)를 한 줄로 고정한다.
2. PoC: Tiptap 또는 Lexical으로 **제한된 블록 세트**(단락, 제목, 목록, 체크, 코드)만 지원하고 **내보내기 → remark 파싱**으로 회귀 테스트한다.
3. 통과 시에만 기본 편집기 교체 범위와 마이그레이션(기존 노트 무손실)을 정한다.
